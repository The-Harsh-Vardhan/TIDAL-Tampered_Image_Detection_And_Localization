# Cross-Document Conflicts

This file lists only material contradictions or decision drift inside the final documentation set.

| Conflicting documents | Conflicting claim | Why it matters | Recommended resolution |
|---|---|---|---|
| `04_Model_Architecture.md` vs `05_Training_Pipeline.md` | `04` defines the baseline as a plain `smp.Unet(...)`, while `05` optimizes `model.unet.encoder`, `model.unet.decoder`, and `model.unet.segmentation_head`. | The training code will break if copied directly, because plain SMP U-Net exposes `model.encoder`, `model.decoder`, and `model.segmentation_head`. | Standardize on one model API. Either train plain `smp.Unet` directly or define a wrapper class and update every snippet to match it. |
| `05_Training_Pipeline.md` vs `10_Project_Timeline.md` and `11_Final_Submission_Checklist.md` | `05` makes `CosineAnnealingWarmRestarts` part of the default training setup, while `10` and `11` treat the scheduler as Stage 2 work. | The baseline training spec is not stable. An implementer cannot tell whether the scheduler is mandatory or optional. | Move the scheduler into one category only. The cleaner choice is baseline = no scheduler required, scheduler = optional upgrade. |
| `03_Data_Pipeline.md` vs `10_Project_Timeline.md` and `11_Final_Submission_Checklist.md` | `03` includes `RandomBrightnessContrast` and `HueSaturationValue` in the default training transform, while `10` and `11` frame additional augmentation as later-stage work. | This changes the real baseline and makes ablation results hard to interpret. | Define one MVP transform set and keep photometric extras in a clearly marked optional block. |
| `04_Model_Architecture.md` and `06_Evaluation_Methodology.md` vs `10_Project_Timeline.md` and `11_Final_Submission_Checklist.md` | `04` and `06` say the decision threshold must be chosen on the validation set, but `10` frames threshold calibration as Stage 2 optimization and `11` partly treats it as an addition rather than a baseline decision. | Threshold choice is part of the evaluation protocol, not a cosmetic improvement. This ambiguity can lead to default `0.5` use in some runs and tuned thresholds in others. | Make threshold policy baseline: choose on validation, lock it before test, and mention Stage 2 only for richer threshold analysis plots. |
| `07_Visual_Results.md` vs `Assignment.md` | `07` centers the main comparison grid on a predicted heatmap, while the assignment asks for a predicted output mask and an overlay. | The required deliverable is a predicted mask, not only a probability heatmap. | Show binary predicted mask as a primary panel. Keep the heatmap as optional supplementary visualization. |
| `01_Assignment_Overview.md` vs `09_Engineering_Practices.md` | `01` says no external services are required for the core submission, while `09` assumes Kaggle API setup and Google Drive checkpointing. | The implementation path described in the docs does rely on external services for the recommended workflow. | Rephrase `01` to say no external serving or deployment platform is required, while Kaggle and optional Drive are part of the practical notebook workflow. |
| `04_Model_Architecture.md` vs `10_Project_Timeline.md` | `04` presents `resnet34`, `efficientnet-b0`, and `efficientnet-b1` as baseline encoder options, while `10` hard-codes ResNet34 in Stage 1. | The baseline encoder choice is not locked, which weakens reproducibility and ablation tracking. | Pick one baseline encoder, preferably ResNet34 or EfficientNet-B0, and move the others into optional comparisons. |
